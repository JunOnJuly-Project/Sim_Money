# ADR-000: 프로젝트 엔드게임 — 개인 전용 실거래·자동매매 시스템

- 상태: Accepted
- 날짜: 2026-04-07
- 결정자: 프로젝트 오너 (본인)

## 컨텍스트

Sim_Money 는 표면적으로는 "비슷하거나 정반대 종목을 찾아주는 웹앱" 이지만,
실제 장기 목표는 **본인 전용 실거래·자동매매 시스템** 이다.
이 엔드게임을 지금(Phase 0) 명시적으로 선언하지 않으면 M1 단계에서 내리는
아키텍처 결정들이 Stage 3 로 가는 길을 막을 위험이 있다.

특히 다음이 문제가 된다:
- UI 와 로직이 뒤섞인 프로토타입 코드는 실거래 경로에서 재사용 불가
- 단일 공식 하드코딩은 매매 전략 교체를 막는다
- 거래 도메인(portfolio, signal, trading, risk)을 나중에 추가하면 기존 모듈과 결합되어 경계가 무너진다
- 실거래 단계의 법적/운영 리스크를 M1 부터 인지해야 한다

## 결정

**1년 내 Stage 3(실거래) 에 도달하는 것을 프로젝트 엔드게임으로 공식 선언한다.**

3단 로켓으로 진행한다:

1. **Stage 1 (MVP, 3~4주)**: 주식 유사도 탐색 + 공식 플레이그라운드 UI
2. **Stage 2 (3~6개월)**: 백테스트 엔진, 공적분, 신호 생성, 페이퍼 트레이딩
3. **Stage 3 (6~12개월)**: 한국투자증권 오픈API 연동, 소액 실거래 → 본격 운용

부가 제약:
- **사용자 범위**: 본인 1인 전용. 제3자 배포·공개 금지 (아이디어 공유는 허용).
- **법률 검토 게이트**: Stage 3 진입 이전 필수. 투자자문/일임/금투업 해당 여부, 개인 자동매매 법적 경계, 한투 API 약관.
- **고지 문구**: 모든 UI 화면에 "개인 사용 목적, 투자 권유 아님" 상시 표시.

## 결과

### 긍정적
- 모든 M1 아키텍처 결정에 일관된 제약 조건이 생긴다: "Stage 3 경로를 막지 않는가?"
- 거래 도메인을 M1 에서 **빈 스켈레톤** 으로 선제 배치할 명분이 생긴다(→ ADR-001).
- 공식을 Strategy 패턴으로 강제할 근거가 된다(→ ADR-002).
- 법률 리스크를 코드가 아닌 프로세스 게이트로 관리한다.

### 부정적 / 트레이드오프
- M1 에 아키텍처 오버헤드가 추가된다 (L3 스켈레톤, import-linter).
- "그냥 재미있는 웹앱" 정도로 끝낼 수 있는 선택지를 포기한다.
- Stage 3 도달 실패 시에도 L3 구조 유지 비용은 이미 지불된다.

### 강제 불변식
- **M1 의 어떤 코드도 Stage 3 경로를 막아서는 안 된다.**
- **L3 도메인(`portfolio`, `signal`, `backtest`, `trading`, `risk`) 스켈레톤은 M1 필수 산출물이다.**
- **실거래 관련 모든 실제 로직은 법률 검토 게이트 통과 이전까지 작성 금지.**
